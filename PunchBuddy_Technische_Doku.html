PunchBuddy

Automatisierungs-App für Pro Tools Punch-In, Transport, Export und Avid Interplay.
Menüleisten-App (macOS) mit Stream-Deck-Integration, Vocaster-Steuerung und Loudness-Normalisierung nach EBU R128.

v 1.2  ·  Erstellt von Jens Rühl  ·  06.2026

1 · Einstellungsmenüs

Alle Einstellungen werden unter ~/.punchbuddy/settings.json gespeichert und beim nächsten Start automatisch geladen. Das Fenster öffnet sich über Menüleiste → Einstellungen. Änderungen werden erst nach dem Klick auf Speichern aktiv.

Tab: Spuren (Profil A & B)

PunchBuddy kennt zwei unabhängige Spurprofile. Profil A wird über den Haupt-Trigger ausgelöst, Profil B über den zweiten Trigger (Menü oder Webtrigger /trigger2). Damit lassen sich z.B. zwei verschiedene Aufnahme-Setups schnell umschalten – etwa Sprecher 1 vs. Sprecher 2.

FeldSettings-KeyBeschreibung
Record-Spuren AtracksSpurnamen (exakt wie in der PT-Session), die beim Profil-A-Trigger record-enabled werden. Komma-getrennte Liste im UI.
Monitor-Spuren Amonitor_tracksSpuren, die beim Play-Trigger (Input-Monitor) auf Eingang geschaltet werden. Nicht beim Punch-In – das übernimmt Pro Tools selbst.
Record-Spuren Btracks_bWie Record-Spuren A, aber für Profil B.
Monitor-Spuren Bmonitor_tracks_bMonitor-Pendant zu Profil B.

Start-Timecodeexport_start_tcTimecode, auf den der PT-Cursor mit Goto Start (Webtrigger /start) gesetzt wird. Format: HH:MM:SS:FF.
Extend Countextend_countLegacy-Wert für die Anzahl Spuren unter der Videospur. Die Selektion wird heute via PTSL (extend_selection_to_target_tracks) auf die konkreten Export-Spuren ausgedehnt – dieser Zähler hat nur noch historische Bedeutung (Standard: 7).
Spurname-Matching: PunchBuddy vergleicht die eingetragenen Namen exakt (case-sensitive) mit den Spurnamen aus dem PTSL-Track-Cache. Tippfehler oder abweichende Schreibweise führen dazu, dass die Spur nicht gefunden wird – der Log zeigt welche Spuren fehlen.

Tab: Presets

Es gibt 8 Presets, die jeweils einen vollständigen Arbeitszustand speichern. Ein Preset kann manuell über das Tab geladen werden oder per Webtrigger /preset/1 bis /preset/8.

Jeder Preset-Eintrag speichert: Record-Spuren A/B, Monitor-Spuren A/B, Export-Spuren, Play-Monitor-Spuren, Lautheits-Spuren sowie alle Bool/String/Int/Float-Einstellungen wie loudness_enabled, interplay_enabled, target_lufs usw.

Tipp: Presets können über Companion oder Stream Deck geladen werden. So lassen sich verschiedene Produktionsformate (Nachrichtensendung, Magazin, …) mit einem Tastendruck komplett umschalten.

2 · Technische Abläufe

Startup

1
Bundle-Name und Prozessname setzen Vor dem Import von rumps/AppKit wird CFBundleName = 'PunchBuddy' ins InfoDictionary geschrieben und NSProcessInfo.setProcessName_ aufgerufen, damit im Dock und Aktivitätsmonitor nicht „Python" erscheint.
2
Logging & Crash-Diagnose initialisieren punchbuddy/log.py erstellt ~/.punchbuddy/PunchBuddy.log, öffnet PunchBuddy_crash.log für faulthandler (SIGSEGV/SIGABRT) und hängt threading.excepthook / sys.excepthook ein.
3
Settings laden & Migration ~/.punchbuddy/settings.json wird per Deep-Merge mit den Default-Werten kombiniert. Falls ein alter ~/.autopunchin/-Ordner existiert, wird er einmalig migriert.
4
rumps App-Loop startet Die Menüleisten-App (PunchBuddyApp) wird instanziiert. HTTP-Server, PTSL-KeepAlive und Vocaster-Timer werden als Daemon-Threads/Timers gestartet.
5
Startup-Pre-Roll-Check (Hintergrund-Thread, 2s Delay) Pre-Roll wird auf AUS gestellt (PTSL), der Track-Cache befüllt. Beides passiert im Hintergrund, damit der Start-Up nicht blockiert.
6
Vocaster-Initialisierung USB-Geräteerkennung via libusb. Falls ein Vocaster gefunden wird: 48V ein (optional) + gespeichertes Routing anwenden (optional) + Menü-Einträge hinzufügen.

Punch-In (Record) Record

Ausgelöst via Menü, Stream Deck (/trigger oder /trigger2) oder Webtrigger. Läuft vollständig in einem Daemon-Thread, damit die UI reaktionsfähig bleibt.

1
Guard: bereits laufend? state.running_lock verhindert parallele Aufnahmen. Wenn schon eine läuft, wird der Trigger ignoriert.
2
Track-Cache abfragen (PTSL) Spurnamen aus dem Session-Cache (einmalig beim Start geladen). Falls leer: frischer PTSL-Call track_list(). Spuren aus der Settings-Liste werden gegen den Cache gematcht.
3
Pre-Roll EIN (PTSL, 3 Versuche) ensure_preroll_on() liest den aktuellen Pre-Roll-Status via GetTimelineSelection, setzt ihn mit SetTimelineSelection(pre_roll_enabled=TB_True). Fallback auf Cmd+K (CGEvent) wenn PTSL scheitert.
4
Record-Enable (PTSL, 3 Versuche, 15s Timeout je) set_track_record_enable_state(tracks, True). 15s Timeout wegen möglicher NEXIS-Schreiblast. Bei Fehler: Warnung im Log, Ablauf wird trotzdem fortgesetzt.
5
Transport-Arm + Play (PTSL) Erst wird geprüft ob der Transport schon armed ist (transport_armed()). Falls nicht: toggle_record_enable() → toggle_play_state(). Pro Tools startet die Aufnahme.
6
Warten bis Transport läuft (max 5s) Polling auf transport_state() alle 50ms bis TS_TransportRecording oder TS_TransportPlaying.
7
Warten bis Transport stoppt (max 10 Minuten) Polling alle 300ms. Bei TS_TransportIsStopping (PT schreibt auf NEXIS) wird auf 3s-Interval umgeschaltet, um PTSL-Last zu reduzieren. Nach 10 konsekutiven Timeouts: Engine-Reset.
8
Transport-Arm AUS + Pre-Roll AUS Transport-Arm wird nur deaktiviert wenn er von PunchBuddy aktiviert wurde. Pre-Roll wird mit restore_preroll() zurückgesetzt.

Play & Play Custom Play

Play (Input Monitor) liest den aktuellen Transport-Status. Wenn PT läuft → Stop-Pfad. Wenn PT gestoppt → Start-Pfad: Input-Monitor der konfigurierten (oder selektierten) Spuren einschalten, dann Playback starten. Beim Stop werden die Monitor-Spuren wieder deaktiviert.

Play Custom funktioniert wie Play, setzt aber vor dem Start definierte Mute-Zustände auf zwei konfigurierbaren Spuren (typisch: KH2 muten, Studio-Abhöre aktiv lassen). Beim Stop werden diese Mutes wieder zurückgesetzt.

Stop Transport

Der Stop-Befehl ist explizit von state.running entkoppelt – er funktioniert auch während einer laufenden Aufnahme. Er verwendet einen eigenen _stop_lock (verhindert Doppel-Stop, blockiert aber keinen parallelen Punch-In). toggle_play_state() entspricht der Leertaste in PT.

Audio verschieben (Spur-Move) Edit

Verschiebt das gesamte Material der Quell-Spuren auf die Ziel-Spuren – an identischer Zeitposition – komplett über PTSL cut/paste. Ausgelöst über Menü „Audio verschieben" oder Webtrigger /move_audio (run_move_audio() in transport.py).

1
Edit-Selection-Kontext aktivieren Über _protools_selection_context: Selector-Tool wählen und „Link Timeline and Edit Selection" + „Link Track and Edit Selection" einschalten. Nur damit steuern select_tracks_by_name / set_timeline_selection die Edit-Selection, auf der cut/paste arbeiten. Originaleinstellungen werden danach wiederhergestellt.
2
Materialbereich ermitteln (PTSL) Pro Quell-Spur select_all_clips_on_track + get_timeline_selection; frühester Start und spätestes Ende ergeben den Bereich. Vor jeder Spur wird die Selektion auf Null-Länge zurückgesetzt, damit leere Spuren (stale-Selektion) korrekt übersprungen werden.
3
Quell-Spuren + Bereich ausschneiden (PTSL) select_tracks_by_name(src) + set_timeline_selection(start, end) spannt die Edit-Selection über alle Quell-Spuren, dann cut(). Settling-Delays zwischen den Schritten, da PT jede Selektionsänderung verarbeiten muss.
4
Auf Ziel-Spuren einfügen (PTSL) select_tracks_by_name(tgt) + Cursor auf den Materialanfang (set_timeline_selection(start, start)), dann paste(). PT mappt die Zwischenablage top→bottom auf die Ziel-Spuren – Quelle/Ziel müssen in gleicher vertikaler Reihenfolge stehen.
Wichtige Erkenntnis: PTSL cut/paste operieren auf der Edit-Selection, nicht auf der Track-/Timeline-Selection. Ohne die beiden Link-Optionen schneidet cut nur eine einzelne Spur aus und paste landet an unerwarteter Stelle. Erst der Selektions-Kontext (wie beim Consolidate-Export) macht den Multi-Track-Move zuverlässig. Live gegen Pro Tools verifiziert: A1/A2 → A3/A4 an exakter Position, Round-Trip stellt den Originalzustand wieder her.

Vollständiger Export-Workflow Export

Wird über Menü oder Webtrigger gestartet. Umfasst je nach Konfiguration: Transport-Stop → Consolidate → WAV → Loudness → AAF → Interplay Export. Die gesamte Spur- und Selektionssteuerung läuft über PTSL – es werden dafür keine simulierten Tastendrücke verwendet.

1
Transport stoppen falls aktiv (PTSL) neu 06.2026 _ensure_transport_stopped() liest transport_state(). Läuft PT noch (Record/Play), wird toggle_play_state() (Stop) gesendet, auf bestätigten Stillstand gewartet und PT 2 s zur Datei-Finalisierung gegeben. Steht der Transport bereits, läuft der Export sofort weiter. Schützt vor Session-Schäden, wenn der User den Export ohne vorheriges Stop auslöst. Läuft in allen fünf Export-Funktionen.
2
Videospur auto-erkennen (PTSL) _detect_video_track(): sucht Spur mit type=4 im PTSL-Track-Cache. Fallback auf den in den Settings eingetragenen Namen.
3
Spuren einblenden + Video-Ende ermitteln (PTSL) set_track_hidden_state(tracks, False) blendet versteckte Export-Spuren ein. select_all_clips_on_track(video_track) + get_timeline_selection() liefern das Out-Time (Video-Ende).
4
Spuren selektieren & Selektion ausdehnen (PTSL) select_tracks_by_name(export_tracks) markiert die Export-Spuren, set_timeline_selection(in_time, out_time) setzt den Zeitbereich, extend_selection_to_target_tracks(export_tracks) dehnt die Edit-Selection auf alle Zielspuren aus. Alles via PTSL – keine Tastatursimulation.
5
Consolidate (PTSL) + auf Abschluss warten engine.consolidate_clip() löst das Consolidate via PTSL aus. Das Warten auf Abschluss nutzt AppleScript: _wait_for_consolidate_window_gone() prüft die PT-Fenstertitel (Wortanfang-Match auf „Consolidating"/„Zusammenführen"). Timeout wird aus der Selektionslänge berechnet.
6
WAV-Datei-Stabilität abwarten _wait_for_consolidated_files(): prüft alle 100ms ob die WAV-Dateien im Audio Files/-Ordner stabil sind (Größe 500ms unverändert). Spuren ohne Material blockieren nicht.
7
Loudness-Normalisierung (optional) Für jede Spur in loudness_tracks: WAV einlesen (soundfile), LUFS messen (pyloudnorm), Gain-Korrektur berechnen, True-Peak-Limiter anwenden, Datei überschreiben. PT wird danach aufgefordert neu einzulesen.
8
AAF-Export (optional) Via AppleScript-UI-Automation im PT-Exportdialog. Embedded oder Reference je nach Konfiguration.
9
Interplay-Export (optional) Siehe eigenen Abschnitt unten.

Interplay Export Interplay

Vollständig via UI-Automation (AppleScript + CGEvent auf den Prozessen Pro Tools und interplayAccess). PTSL bietet dafür keine API.

1
PT aktivieren, Export-Dialog öffnen AppleScript aktiviert Pro Tools, öffnet Clip → Export to MediaCentral. PT zeigt einen modalen Dialog mit Fortschrittsbalken.
2
PT-Log via kqueue überwachen Das PT-Log (~/Library/Logs/Avid/Pro_Tools_*.txt) wird per kqueue (VNODE-Event) live beobachtet. Jede neue Zeile wird auf SendDialogOnScreen … error msg = … geprüft. „successfully" → Erfolg; anderer Text → Fehler.
3
PT-Erfolgsdialog bestätigen AppleScript sendet Return auf den PT-OK-Dialog. _wait_for_pt_modal_dismissed() wartet via kqueue bis PT den Dialog-Schließ-Event ins Log schreibt.
4
Interplay Access aktivieren + Workspace navigieren AppleScript aktiviert interplayAccess. Tab-Tastendrücke navigieren den Workspace-Baum zur konfigurierten Anzahl von Schritten.
5
Umbenennen (optional) F2 öffnet das Rename-Feld. Name wird ausgelesen (Cmd+A → Cmd+C), nach Vorgaben manipuliert (Trim + Prefix + Suffix), zurückgeschrieben und mit Enter bestätigt. Das Feld bleibt dabei offen – kein doppeltes Öffnen.

Interplay Import Interplay

Öffnet in Interplay Access eine Sequenz und übergibt sie an Pro Tools. Optional wird die aktuelle PT-Session vorher automatisch geschlossen (import_close_session). PT bestätigt den Import per Dialog-Fenster, das per AppleScript gelesen wird.

Lautheitsnormalisierung (EBU R128) Loudness

Läuft nach dem Consolidate, vor dem AAF-Export. Alle Schritte laufen im Arbeitsspeicher – die WAV-Datei wird erst am Ende überschrieben.

  1. Neueste <Spurname>*.wav im Audio Files/-Ordner finden (nach mtime)
  2. Audio laden mit soundfile.read()
  3. Integrierte Lautheit messen mit pyloudnorm.Meter.integrated_loudness()
  4. Gain-Differenz berechnen: gain_db = target_lufs − current_lufs
  5. Gain anwenden: data *= 10^(gain_db/20)
  6. True-Peak-Limiter: Peaks über max_truepeak werden zurückgerechnet
  7. Datei überschreiben mit soundfile.write()
  8. Pro Tools: Clip → Refresh via AppleScript damit PT die neue Datei liest

Vocaster USB-Steuerung Vocaster

PunchBuddy kommuniziert direkt per USB (HID) mit dem Focusrite Vocaster. Die vocaster_control.VocasterUSB-Klasse kapselt die USB-Transaktionen. Die gesamte Logik läuft im Main-Thread via rumps.Timer (alle 400ms), weil AppKit-Fenster nur dort bedient werden dürfen.

FunktionAblauf
Geräteerkennungdetect_vocaster() sucht per Vendor/Product-ID über libusb nach einem angeschlossenem Vocaster One oder Two.
48V PhantomspeisungEin einzelner USB HID-Report an Registernummer für 48V-Bit. Funktioniert sofort, kein Dialog.
AutogainUSB-Command startet den Autogain-Prozess im Gerät. Ein AutogainWindow (natives AppKit-Fenster) zeigt Fortschritt und Ergebnis. Der drive()-Tick pollt alle 400ms den Status über USB aus, bis Success/Fail/Timeout.
Routingcapture_routing_now() liest das aktuelle MUX-Routing aller Kanäle per USB aus und speichert es als JSON. apply_saved_routing() schreibt es zurück.

Module & Abhängigkeiten

Die Abhängigkeiten sind strikt hierarchisch – kein Modul importiert von einem „höheren". Das verhindert zyklische Imports.

state.py

Blatt-Modul. Kein Import aus dem Package. Enthält nur running, running_lock, import_running, export_running, app_ref, Icon-Konstanten.

config.py

Kritische Regel: Alle AppKit/NSStatusItem-Aufrufe müssen im Main-Thread erfolgen. _dispatch_main(fn) in uikit.py verwendet AppHelper.callAfter() (performSelectorOnMainThread:), das threadsicher den Run-Loop aufweckt. NSRunLoop.performBlock_() ist nicht threadsicher und darf nicht verwendet werden.

PTSL / gRPC Core

PTSL (Pro Tools Scripting Layer) ist eine gRPC-basierte API von Avid, die auf Port 31416 lauscht. Das Python-Package ptsl wraps die Proto-Definitionen. PunchBuddy verwaltet eine Singleton-Instanz (_engine_instance) mit automatischem Reconnect.

gRPC-Deadline-Wrapper (engine.py:_install_grpc_deadline)

Das ptsl-Package reicht keinen Timeout an den gRPC-Stub durch. PunchBuddy patcht deshalb raw_client.SendGrpcRequest mit einem Wrapper, der timeout=_current_grpc_deadline() setzt. Der Wert kommt aus Thread-Local-Storage (_grpc_deadline_tls) und wird von _ptsl_call() vor jedem Call gesetzt.

_ptsl_call() – serialisierter Aufruf

def _ptsl_call(fn, *args, label="", timeout=15.0):
    # Exklusiver Lock: verhindert parallele gRPC-Calls (NEXIS-Delay-Schutz)
    if not _ptsl_lock.acquire(timeout=6.0):
        return False, None

    _grpc_deadline_tls.value = timeout  # Deadline für diesen Call
    try:
        return True, fn(*args)
    except grpc.RpcError as e:
        if e.code() == grpc.StatusCode.DEADLINE_EXCEEDED:
            ...  # Timeout → Engine verwerfen
        _reset_engine()
        return False, e
    finally:
        _ptsl_lock.release()

Track-Cache

Spurnamen ändern sich während einer Session nicht. _get_cached_track_names() lädt sie einmalig (beim Start oder beim ersten Record-Trigger) und hält sie im Speicher. Der Cache wird bei Engine-Reset / Session-Wechsel via _invalidate_session_tracks() geleert. run_play() liest Tracks weiterhin live, da die aktuelle Selektion in Echtzeit benötigt wird.

AppKit & Main-Thread-Dispatch AppKit

# uikit.py – threadsicherer Dispatch auf Main-Thread
def _dispatch_main(fn):
    def _safe():
        try:
            fn()
        except Exception:
            logging.exception("_dispatch_main: Fehler im Main-Thread-Block")
    try:
        from PyObjCTools import AppHelper
        AppHelper.callAfter(_safe)   # = performSelectorOnMainThread:
    except Exception:
        pass

Warum nicht NSRunLoop.performBlock_()? – Diese Methode ist nicht threadsicher und weckt den Run-Loop nicht auf. Calls aus Worker-Threads können silently verloren gehen oder zu Race-Conditions führen. AppHelper.callAfter() verwendet intern performSelectorOnMainThread:withObject:waitUntilDone: (wait=False) und ist das korrekte Pattern.

CGEvent – Tastatursteuerung Input

Für die wenigen Aktionen ohne PTSL-API werden Tastenanschläge als CGEvents direkt an die Pro Tools PID geschickt – das funktioniert auch wenn PT nicht im Vordergrund ist. In der Praxis bleibt davon nur der Pre-Roll-Fallback (Cmd+K), falls das Setzen via PTSL SetTimelineSelection dreimal scheitert. Die Selektions-Erweiterung beim Export läuft nicht über Tastatur, sondern über PTSL extend_selection_to_target_tracks(); die alte Shift+Ö-Hilfsfunktion (_send_shift_oe) ist im aktuellen Code-Pfad nicht mehr aktiv.

# keys.py
def _send_key_to_app(pid, vk, cmd=False, shift=False):
    src = Quartz.CGEventSourceCreate(kCGEventSourceStateHIDSystemState)
    ev_dn = Quartz.CGEventCreateKeyboardEvent(src, vk, True)
    if cmd:
        Quartz.CGEventSetFlags(ev_dn, kCGEventFlagMaskCommand)
    Quartz.CGEventPostToPid(pid, ev_dn)
    Quartz.CGEventPostToPid(pid, Quartz.CGEventCreateKeyboardEvent(src, vk, False))

VK-Konstanten (macOS Hardware-Codes, layout-unabhängig): _VK_K=40 (Cmd+K = Pre-Roll-Fallback), _VK_F2=120 (Interplay Rename), _VK_ESC=53, _VK_RETURN=36. CGEvents kommen außerdem in der Interplay-UI-Automation zum Einsatz (F2/Tab/Cmd+A/C/V), da Interplay Access keine PTSL-API hat.

AppleScript / UI-Automation UI

AppleScript via osascript (subprocess) wird für Aktionen genutzt, die keine PTSL-API haben:

  • Fenstertitel von PT auslesen (name of every window) – z.B. um auf das Ende des Consolidate-Vorgangs zu warten
  • PT-Exportmenü auslösen (Clip → Export to MediaCentral). Consolidate selbst läuft via PTSL consolidate_clip(), nicht über AppleScript.
  • Interplay Access steuern (Tab-Navigation, F2 Rename, Cmd+C/V)
  • Static-Text aus PT-Dialogfenstern lesen (Export-Ergebnis)
  • App aktivieren (tell application "interplayAccess" to activate)

Alle AppleScript-Calls werden über _run_applescript(script, timeout=30) gekapselt, das einen subprocess.run()-Prozess startet. Bei Timeouts gibt die Funktion einen Fehler-Returncode zurück.

Accessibility-Berechtigung: AppleScript-Zugriff auf System Events erfordert, dass PunchBuddy (oder Python) unter Systemeinstellungen → Datenschutz → Bedienungshilfen freigeschaltet ist.

kqueue – Dateiüberwachung macOS

kqueue ist der macOS-Kernel-Event-Benachrichtigungsmechanismus. PunchBuddy benutzt ihn um das PT-Log-File zu überwachen, ohne Polling:

# export.py – Vereinfacht
fd = os.open(log_path, os.O_RDONLY | os.O_NONBLOCK)
kq = select.kqueue()
ke = select.kevent(fd,
    filter=select.KQ_FILTER_VNODE,
    flags=select.KQ_EV_ADD | select.KQ_EV_ENABLE | select.KQ_EV_CLEAR,
    fflags=select.KQ_NOTE_WRITE | select.KQ_NOTE_EXTEND)

while time.time() < deadline:
    events = kq.control([ke], 4, min(remaining, 2.0))  # blockiert bis Event
    if events:
        # Neue Bytes lesen, auf Marker prüfen
        ...

Vorteile gegenüber Polling: sofortige Reaktion bei Datei-Event (kein festes 0.3s-Intervall), keine CPU-Last zwischen Schreibvorgängen. Fallback auf 0.3s-Polling wenn kqueue aus irgendeinem Grund nicht verfügbar ist.

HTTP-Server / Webtrigger HTTP

PunchBuddy startet beim Launch einen eingebetteten ThreadingMixIn + TCPServer. Pro Request wird sofort 200 OK geantwortet, die eigentliche Funktion läuft in einem Daemon-Thread (_fire(fn)). So antwortet der Stream Deck binnen Millisekunden, auch wenn der Export Sekunden dauert.

Security: Bei Bind auf 0.0.0.0 (LAN) wird automatisch ein kryptografisch sicheres Token (secrets.token_urlsafe(16)) erzeugt und gespeichert. Requests ohne korrektes Token bekommen 401 Unauthorized. Token-Vergleich via hmac.compare_digest() (timing-attack-sicher).

Settings-System Config

Einstellungen werden als JSON unter ~/.punchbuddy/settings.json gespeichert. Beim Laden wird ein rekursiver Deep-Merge mit den Default-Werten durchgeführt: Neu hinzugekommene Keys aus dem Code werden automatisch ergänzt, bestehende Nutzerwerte bleiben erhalten.

def _deep_merge(default, override):
    """Neue Keys aus default werden ergänzt; override gewinnt bei Konflikten."""
    if isinstance(default, dict) and isinstance(override, dict):
        out = dict(default)
        for k, v in override.items():
            out[k] = _deep_merge(default[k], v) if k in default else v
        return out
    return override

Preset-Einträge bekommen zusätzlich ein Template-Merge (_PRESET_TEMPLATE), damit neu hinzugekommene Preset-Keys auch in gespeicherten alten Presets erscheinen.

Crash-Diagnose Debug

Drei Ebenen fangen Fehler ab, die sonst spurlos verschwinden würden:

MechanismusDateiWas er fängt
faulthandlerlog.pyNative Crashes (SIGSEGV, SIGABRT) – schreibt C-Level-Stacktraces aller Python-Threads in PunchBuddy_crash.log. Unverzichtbar bei AppKit/ObjC-Fehlern.

Das Diagnose-Script collect_diagnostics.py (bzw. PunchBuddy_Diagnose.app aus dem DMG) sammelt alle relevanten Logs, PT-Crash-Reports, System-Info und den faulthandler-Output in eine einzige Textdatei auf dem Desktop.

Wichtig für Build: Der Universal2-Build (build_dmg.sh) benötigt zwei Python-Installationen – eine native ARM64 und eine x86_64 – mit jeweils eigenen Paketen (numpy, scipy etc.). Der Intel-Build (build_dmg_intel.sh) läuft mit dem Framework-Python 3.14 (x86_64) und bündelt libusb automatisch. Für den Studio-Rechner (Intel Mac) ist der Intel-Build der korrekte.
