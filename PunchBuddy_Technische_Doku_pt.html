PunchBuddy

Aplicação de automação para Pro Tools Punch-In, transporte, exportação e Avid Interplay.
Aplicação na barra de menus (macOS) com integração Stream Deck, controlo do Vocaster e normalização de loudness segundo EBU R128.

v 1.3  ·  Criado por Jens Rühl  ·  06.2026

1 · Menus de configuração

Todas as configurações são guardadas em ~/.punchbuddy/settings.json e carregadas automaticamente no arranque seguinte. A janela abre-se através de Barra de menus → Configurações. As alterações só ficam ativas depois de clicar em Guardar.

Separador: Faixas (Perfil A & B)

O PunchBuddy conhece dois perfis de faixas independentes. O Perfil A é acionado pelo gatilho principal, o Perfil B pelo segundo gatilho (menu ou webtrigger /trigger2). Isto permite, por exemplo, alternar rapidamente entre duas configurações de gravação diferentes – como Locutor 1 vs. Locutor 2.

Secção Stream Deck

O PunchBuddy traz consigo o plugin do Stream Deck e os lançadores de botões diretamente (agrupados na app). Três botões:

BotãoEfeito
Instalar pluginAbre o com.punchbuddy.control.streamDeckPlugin fornecido – o Stream Deck mostra o seu próprio diálogo de instalação. Depois fica disponível a ação „Comando PunchBuddy" com um dropdown de comandos.
Guardar plugin (Downloads)Copia o ficheiro do plugin para ~/Downloads e mostra-o no Finder (para partilhar/instalar manualmente).
Criar lançadores de botõesCria por cada comando um .app minúsculo em ~/Applications/PunchBuddy Launchers, que envia o comando via nc -U ao Unix-Socket configurado. Utilizável no Stream Deck através da ação incorporada „Sistema → Abrir" – sem instalação de plugin.
Porquê dois caminhos? O plugin (uma ação, comando por dropdown) é cómodo, mas requer Stream Deck ≥ 6.5. Os lançadores funcionam com qualquer versão do Stream Deck, totalmente sem instalação de plugin. Ambos atingem o mesmo socket local – completamente sem rede.

Separador: Webtrigger

Todas as funções acionáveis passam por uma API de comandos central (command_dispatch). Esta é servida por dois transportes:

  • Servidor HTTP (ThreadingMixIn + TCPServer, porta padrão 8899) – para Stream Deck, Companion ou quaisquer clientes HTTP.
  • Unix-Domain-Socket (/tmp/punchbuddy.sock) – completamente sem rede (sem TCP/Loopback) e, por isso, em princípio não detetável por filtros de rede como o Microsoft Defender. Protegido por permissões de ficheiro 0600 (apenas o utilizador com sessão iniciada).
CampoChave de configuraçãoDescrição
Portahttp_portPorta TCP do servidor HTTP (padrão: 8899). As alterações só ficam ativas após reiniciar. Se a porta estiver ocupada, é escolhida automaticamente uma porta livre.
Bind-Hosthttp_bind_host127.0.0.1 = apenas local (seguro, sem necessidade de token). Qualquer outro IP = acessível na rede → é gerado automaticamente um token.
Tokenwebtrigger_tokenToken de autenticação opcional (apenas HTTP). Num bind de rede é gerado automaticamente. Tem de ser passado como ?token=… ou no cabeçalho X-Auth-Token.
Unix-Socket ativounix_socket_enabledAtiva/desativa o socket sem rede (padrão: ligado).
Caminho do socketunix_socket_pathCaminho do Unix-Socket (padrão: /tmp/punchbuddy.sock). Protocolo: uma linha <befehl> [arg], resposta OK … / ERR …. Teste: printf 'play' | nc -U /tmp/punchbuddy.sock.

Separador: Presets

Existem 8 presets, cada um guardando um estado de trabalho completo. Um preset pode ser carregado manualmente pelo separador ou por webtrigger /preset/1 a /preset/8.

Cada entrada de preset guarda: faixas de gravação A/B, faixas de monitorização A/B, faixas de exportação, faixas de monitorização Play, faixas de loudness, bem como todas as configurações Bool/String/Int/Float como loudness_enabled, interplay_enabled, target_lufs, etc.

Dica: Os presets podem ser carregados através do Companion ou do Stream Deck. Assim é possível alternar completamente entre diferentes formatos de produção (noticiário, revista, …) com uma só tecla.

2 · Fluxos técnicos

Inicialização

1
Definir nome do bundle e nome do processo Antes da importação de rumps/AppKit, é escrito CFBundleName = 'PunchBuddy' no InfoDictionary e é chamado NSProcessInfo.setProcessName_, para que no Dock e no Monitor de Atividade não apareça „Python".
2
Inicializar logging & diagnóstico de crashes punchbuddy/log.py cria ~/.punchbuddy/PunchBuddy.log, abre PunchBuddy_crash.log para o faulthandler (SIGSEGV/SIGABRT) e instala threading.excepthook / sys.excepthook.
3
Carregar configurações & migração ~/.punchbuddy/settings.json é combinado com os valores por defeito através de deep-merge. Se existir uma pasta antiga ~/.autopunchin/, é migrada uma única vez.
4
O loop da app rumps arranca A aplicação da barra de menus (PunchBuddyApp) é instanciada. O servidor HTTP, o PTSL-KeepAlive e o timer do Vocaster são iniciados como threads/timers daemon.
5
Verificação de Pre-Roll no arranque (thread em segundo plano, atraso de 2s) O Pre-Roll é colocado em OFF (PTSL) e o cache de faixas é preenchido. Ambas as operações ocorrem em segundo plano, para não bloquear o arranque.
6
Inicialização do Vocaster Deteção de dispositivos USB via libusb. Se um Vocaster for encontrado: 48V ligado (opcional) + aplicar routing guardado (opcional) + adicionar entradas de menu.

Punch-In (Gravação) Record

Acionado via menu, Stream Deck (/trigger ou /trigger2) ou webtrigger. Corre inteiramente numa thread daemon, para que a interface se mantenha responsiva.

1
Guard: já em execução? state.running_lock impede gravações em paralelo. Se já houver uma em curso, o gatilho é ignorado.
2
Consultar o cache de faixas (PTSL) Nomes de faixas do cache da sessão (carregado uma vez no arranque). Se estiver vazio: chamada PTSL nova track_list(). As faixas da lista de configurações são comparadas contra o cache.
3
Pre-Roll LIGADO (PTSL, 3 tentativas) ensure_preroll_on() lê o estado atual do Pre-Roll via GetTimelineSelection e define-o com SetTimelineSelection(pre_roll_enabled=TB_True). Fallback para Cmd+K (CGEvent) se o PTSL falhar.
4
Record-Enable (PTSL, 3 tentativas, timeout de 15s cada) set_track_record_enable_state(tracks, True). Timeout de 15s devido à possível carga de escrita do NEXIS. Em caso de erro: aviso no log, o processo continua mesmo assim.
5
Transport-Arm + Play (PTSL) Primeiro verifica-se se o transporte já está armado (transport_armed()). Se não: toggle_record_enable() → toggle_play_state(). O Pro Tools inicia a gravação.
6
Esperar até o transporte estar a correr (máx. 5s) Polling de transport_state() a cada 50ms até TS_TransportRecording ou TS_TransportPlaying.
7
Esperar até o transporte parar (máx. 10 minutos) Polling a cada 300ms. Em TS_TransportIsStopping (o PT está a escrever no NEXIS) muda-se para um intervalo de 3s, para reduzir a carga PTSL. Após 10 timeouts consecutivos: reset do engine.
8
Transport-Arm DESLIGADO + Pre-Roll DESLIGADO O Transport-Arm só é desativado se tiver sido ativado pelo PunchBuddy. O Pre-Roll é reposto com restore_preroll().

Play & Play Custom Play

Play (Input Monitor) lê o estado atual do transporte. Se o PT estiver a correr → caminho de Stop. Se o PT estiver parado → caminho de início: ligar o Input Monitor das faixas configuradas (ou selecionadas) e depois iniciar o playback. No Stop, as faixas de monitorização são novamente desativadas.

Play Custom funciona como o Play, mas antes do início define estados de mute definidos em duas faixas configuráveis (tipicamente: silenciar KH2, deixar a escuta de estúdio ativa). No Stop, esses mutes são novamente repostos.

Stop Transport

O comando Stop está explicitamente desacoplado de state.running – funciona mesmo durante uma gravação em curso. Usa um _stop_lock próprio (impede o duplo Stop, mas não bloqueia um Punch-In paralelo). toggle_play_state() corresponde à barra de espaço no PT.

Mover áudio (Track-Move) Edit

Move todo o material das faixas de origem para as faixas de destino – na mesma posição temporal – inteiramente via PTSL cut/paste. Acionado pelo menu „Mover áudio" ou pelo webtrigger /move_audio (run_move_audio() em transport.py).

1
Ativar o contexto de Edit-Selection Através de _protools_selection_context: escolher a ferramenta Selector e ligar „Link Timeline and Edit Selection" + „Link Track and Edit Selection". Só assim é que select_tracks_by_name / set_timeline_selection controlam a Edit-Selection sobre a qual cut/paste operam. As configurações originais são restauradas depois.
2
Mover par a par por faixa (src[i] → tgt[i]) A associação é feita explicitamente por índice, não pelo mapeamento top→bottom do PT. Cada faixa é movida individualmente; antes de cada faixa de origem, a seleção é colocada em comprimento zero, para que faixas vazias (seleção obsoleta) sejam corretamente ignoradas.
3
Cortar com base em clips (PTSL) select_all_clips_on_track(src[i]) cria uma Edit-Selection ao nível do clip e depois cut(). O início original do clip é memorizado. Há atrasos de estabilização entre os passos, uma vez que o PT tem de processar cada alteração de seleção.
4
Colar na faixa de destino na mesma posição (PTSL) select_tracks_by_name([tgt[i]]) + cursor no início de clip memorizado (set_timeline_selection(start, start)), depois paste(). Cada faixa fica na sua própria posição original – independentemente do timecode de início.
Dois aprendizados do teste ao vivo: (1) O PTSL cut/paste opera sobre a Edit-Selection – sem as opções de Link, o cut corta apenas uma faixa e o paste fica na posição errada. (2) Uma seleção multipista baseada em intervalo temporal corta de forma pouco fiável quando o material não começa em posição exata de frame (subframes, por exemplo 10:00:00:11.64). Por isso é baseada em clips e faixa a faixa. Verificado contra o Pro Tools: tanto material alinhado ao frame (A1/A2 → A3/A4) como material deslocado por subframe (A5/A6 → A7/A8) é movido para a posição exata; o round-trip repõe o estado original.

Fluxo de exportação completo Export

Iniciado via menu ou webtrigger. Abrange, consoante a configuração: Stop do transporte → Consolidate → WAV → Loudness → AAF → Exportação Interplay. Todo o controlo de faixas e seleção corre via PTSL – para isso não são usadas teclas simuladas.

1
Parar o transporte se estiver ativo (PTSL) novo 06.2026 _ensure_transport_stopped() lê transport_state(). Se o PT ainda estiver a correr (Record/Play), é enviado toggle_play_state() (Stop), espera-se pela paragem confirmada e dão-se 2 s ao PT para finalizar os ficheiros. Se o transporte já estiver parado, a exportação continua de imediato. Protege contra danos na sessão quando o utilizador aciona a exportação sem um Stop prévio. Corre em todas as cinco funções de exportação.
2
Deteção automática da faixa de vídeo (PTSL) _detect_video_track(): procura uma faixa com type=4 no cache de faixas PTSL. Fallback para o nome introduzido nas configurações.
3
Mostrar faixas + determinar o fim do vídeo (PTSL) set_track_hidden_state(tracks, False) mostra faixas de exportação ocultas. select_all_clips_on_track(video_track) + get_timeline_selection() fornecem o Out-Time (fim do vídeo).
4
Selecionar faixas & estender a seleção (PTSL) select_tracks_by_name(export_tracks) marca as faixas de exportação, set_timeline_selection(in_time, out_time) define o intervalo temporal, extend_selection_to_target_tracks(export_tracks) estende a Edit-Selection a todas as faixas de destino. Tudo via PTSL – sem simulação de teclado.
5
Consolidate (PTSL) + esperar pela conclusão engine.consolidate_clip() aciona o Consolidate via PTSL. A espera pela conclusão usa AppleScript: _wait_for_consolidate_window_gone() verifica os títulos das janelas do PT (correspondência no início da palavra para „Consolidating"/„Zusammenführen"). O timeout é calculado a partir do comprimento da seleção.
6
Aguardar a estabilidade dos ficheiros WAV _wait_for_consolidated_files(): verifica a cada 100ms se os ficheiros WAV na pasta Audio Files/ estão estáveis (tamanho inalterado durante 500ms). Faixas sem material não bloqueiam.
7
Normalização de loudness (opcional) Para cada faixa em loudness_tracks: ler o WAV (soundfile), medir LUFS (pyloudnorm), calcular a correção de ganho, aplicar limitador de True-Peak, sobrescrever o ficheiro. Em seguida, o PT é instruído a reler.
8
Exportação AAF (opcional) Via automação de UI com AppleScript no diálogo de exportação do PT. Embedded ou Reference consoante a configuração.
9
Exportação Interplay (opcional) Ver a secção própria abaixo.

Exportação Interplay Interplay

Inteiramente via automação de UI (AppleScript + CGEvent nos processos Pro Tools e interplayAccess). O PTSL não oferece API para isto.

1
Ativar o PT, abrir o diálogo de exportação O AppleScript ativa o Pro Tools e abre Clip → Export to MediaCentral. O PT mostra um diálogo modal com barra de progresso.
2
Monitorizar o log do PT via kqueue O log do PT (~/Library/Logs/Avid/Pro_Tools_*.txt) é observado ao vivo via kqueue (evento VNODE). Cada nova linha é verificada quanto a SendDialogOnScreen … error msg = …. „successfully" → sucesso; outro texto → erro.
3
Confirmar o diálogo de sucesso do PT O AppleScript envia Return ao diálogo OK do PT. _wait_for_pt_modal_dismissed() espera via kqueue até o PT escrever no log o evento de fecho do diálogo.
4
Ativar o Interplay Access + navegar no workspace O AppleScript ativa o interplayAccess. Pressões da tecla Tab navegam na árvore de workspaces até ao número de passos configurado.
5
Renomear (opcional) F2 abre o campo de renomeação. O nome é lido (Cmd+A → Cmd+C), manipulado conforme as regras (Trim + Prefixo + Sufixo), reescrito e confirmado com Enter. O campo permanece aberto – sem abertura dupla.

Importação Interplay Interplay

Abre uma sequência no Interplay Access e entrega-a ao Pro Tools. Opcionalmente, a sessão atual do PT é fechada automaticamente antes (import_close_session). O PT confirma a importação com uma janela de diálogo que é lida via AppleScript.

Normalização de loudness (EBU R128) Loudness

Corre após o Consolidate, antes da exportação AAF. Todos os passos correm em memória – o ficheiro WAV só é sobrescrito no fim.

  1. Encontrar o <Spurname>*.wav mais recente na pasta Audio Files/ (por mtime)
  2. Carregar o áudio com soundfile.read()
  3. Medir a loudness integrada com pyloudnorm.Meter.integrated_loudness()
  4. Calcular a diferença de ganho: gain_db = target_lufs − current_lufs
  5. Aplicar o ganho: data *= 10^(gain_db/20)
  6. Limitador de True-Peak: os picos acima de max_truepeak são recalculados
  7. Sobrescrever o ficheiro com soundfile.write()
  8. Pro Tools: Clip → Refresh via AppleScript para que o PT leia o novo ficheiro

Controlo USB do Vocaster Vocaster

O PunchBuddy comunica diretamente por USB (HID) com o Focusrite Vocaster. A classe vocaster_control.VocasterUSB encapsula as transações USB. Toda a lógica corre no Main-Thread via rumps.Timer (a cada 400ms), porque as janelas AppKit só aí podem ser manipuladas.

FunçãoFluxo
Deteção de dispositivodetect_vocaster() procura, por Vendor/Product-ID via libusb, um Vocaster One ou Two ligado.

Modelo de threading

O PunchBuddy é uma app de Main-Thread único (rumps/AppKit), que delega todas as operações de longa duração para threads daemon.

Regra crítica: Todas as chamadas AppKit/NSStatusItem têm de ocorrer no Main-Thread. _dispatch_main(fn) em uikit.py usa AppHelper.callAfter() (performSelectorOnMainThread:), que acorda o run-loop de forma thread-safe. NSRunLoop.performBlock_() não é thread-safe e não deve ser usado.

PTSL / gRPC Core

PTSL (Pro Tools Scripting Layer) é uma API baseada em gRPC da Avid, que escuta na porta 31416. O pacote Python ptsl encapsula as definições Proto. O PunchBuddy gere uma instância singleton (_engine_instance) com reconexão automática.

Wrapper de deadline gRPC (engine.py:_install_grpc_deadline)

O pacote ptsl não passa um timeout ao stub gRPC. Por isso, o PunchBuddy faz patch de raw_client.SendGrpcRequest com um wrapper que define timeout=_current_grpc_deadline(). O valor vem de Thread-Local-Storage (_grpc_deadline_tls) e é definido por _ptsl_call() antes de cada chamada.

_ptsl_call() – chamada serializada

def _ptsl_call(fn, *args, label="", timeout=15.0):
    # Lock exclusivo: impede chamadas gRPC paralelas (proteção contra atraso do NEXIS)
    if not _ptsl_lock.acquire(timeout=6.0):
        return False, None

    _grpc_deadline_tls.value = timeout  # Deadline para esta chamada
    try:
        return True, fn(*args)
    except grpc.RpcError as e:
        if e.code() == grpc.StatusCode.DEADLINE_EXCEEDED:
            ...  # Timeout → descartar o engine
        _reset_engine()
        return False, e
    finally:
        _ptsl_lock.release()

Cache de faixas

Os nomes das faixas não mudam durante uma sessão. _get_cached_track_names() carrega-os uma única vez (no arranque ou no primeiro gatilho de gravação) e mantém-nos em memória. O cache é limpo num reset do engine / mudança de sessão via _invalidate_session_tracks(). run_play() continua a ler as faixas ao vivo, pois a seleção atual é necessária em tempo real.

AppKit & Dispatch no Main-Thread AppKit

# uikit.py – dispatch thread-safe para o Main-Thread
def _dispatch_main(fn):
    def _safe():
        try:
            fn()
        except Exception:
            logging.exception("_dispatch_main: Fehler im Main-Thread-Block")
    try:
        from PyObjCTools import AppHelper
        AppHelper.callAfter(_safe)   # = performSelectorOnMainThread:
    except Exception:
        pass

Porquê não NSRunLoop.performBlock_()? – Este método não é thread-safe e não acorda o run-loop. Chamadas a partir de worker-threads podem perder-se silenciosamente ou provocar race-conditions. AppHelper.callAfter() usa internamente performSelectorOnMainThread:withObject:waitUntilDone: (wait=False) e é o padrão correto.

CGEvent – Controlo de teclado Input

Para as poucas ações sem API PTSL, os toques de tecla são enviados como CGEvents diretamente para o PID do Pro Tools – o que funciona mesmo quando o PT não está em primeiro plano. Na prática, resta apenas o fallback de Pre-Roll (Cmd+K), caso a definição via PTSL SetTimelineSelection falhe três vezes. A extensão de seleção na exportação não passa pelo teclado, mas por PTSL extend_selection_to_target_tracks(); a antiga função auxiliar Shift+Ö (_send_shift_oe) já não está ativa no caminho de código atual.

# keys.py
def _send_key_to_app(pid, vk, cmd=False, shift=False):
    src = Quartz.CGEventSourceCreate(kCGEventSourceStateHIDSystemState)
    ev_dn = Quartz.CGEventCreateKeyboardEvent(src, vk, True)
    if cmd:
        Quartz.CGEventSetFlags(ev_dn, kCGEventFlagMaskCommand)
    Quartz.CGEventPostToPid(pid, ev_dn)
    Quartz.CGEventPostToPid(pid, Quartz.CGEventCreateKeyboardEvent(src, vk, False))

Constantes VK (hardware codes do macOS, independentes do layout): _VK_K=40 (Cmd+K = fallback de Pre-Roll), _VK_F2=120 (renomear no Interplay), _VK_ESC=53, _VK_RETURN=36. Os CGEvents são também usados na automação de UI do Interplay (F2/Tab/Cmd+A/C/V), uma vez que o Interplay Access não tem API PTSL.

AppleScript / Automação de UI UI

AppleScript via osascript (subprocess) é usado para ações que não têm API PTSL:

  • Ler os títulos de janela do PT (name of every window) – por exemplo, para esperar pelo fim do processo de Consolidate
  • Acionar o menu de exportação do PT (Clip → Export to MediaCentral). O próprio Consolidate corre via PTSL consolidate_clip(), não via AppleScript.
  • Controlar o Interplay Access (navegação por Tab, renomear com F2, Cmd+C/V)
  • Ler o Static-Text das janelas de diálogo do PT (resultado da exportação)
  • Ativar a app (tell application "interplayAccess" to activate)

Todas as chamadas AppleScript são encapsuladas por _run_applescript(script, timeout=30), que inicia um processo subprocess.run(). Em caso de timeouts, a função devolve um código de retorno de erro.

Permissão de Acessibilidade: O acesso do AppleScript a System Events exige que o PunchBuddy (ou o Python) esteja autorizado em Definições do Sistema → Privacidade → Acessibilidade.

kqueue – Monitorização de ficheiros macOS

kqueue é o mecanismo de notificação de eventos do kernel do macOS. O PunchBuddy usa-o para monitorizar o ficheiro de log do PT, sem polling:

# export.py – Simplificado
fd = os.open(log_path, os.O_RDONLY | os.O_NONBLOCK)
kq = select.kqueue()
ke = select.kevent(fd,
    filter=select.KQ_FILTER_VNODE,
    flags=select.KQ_EV_ADD | select.KQ_EV_ENABLE | select.KQ_EV_CLEAR,
    fflags=select.KQ_NOTE_WRITE | select.KQ_NOTE_EXTEND)

while time.time() < deadline:
    events = kq.control([ke], 4, min(remaining, 2.0))  # bloqueia até ao evento
    if events:
        # Ler novos bytes, verificar o marcador
        ...

Vantagens face ao polling: reação imediata a um evento de ficheiro (sem intervalo fixo de 0.3s), sem carga de CPU entre escritas. Fallback para polling de 0.3s se o kqueue não estiver disponível por alguma razão.

API de controlo: Dispatcher + Transportes API

No núcleo está uma tabela de comandos command_dispatch(command, args) → (ok, msg) (_command_table()). UMA entrada por função (record_a/b, play, play_custom, goto_start, move_audio, import, todas as exportações, preset, vocaster_*, ping/status/list). Todos os transportes chamam exclusivamente este dispatcher – assim, HTTP e Socket partilham automaticamente o debounce e a serialização de exportações.

Transporte 1 – HTTP: ThreadingMixIn + TCPServer embebido. Por cada pedido de gatilho responde imediatamente 200 OK, a função continua em fila. Socket-Timeout (30 s) contra clientes lentos. Num bind em 0.0.0.0 (LAN) é gerado automaticamente um token (secrets.token_urlsafe(16)); pedidos sem o token correto recebem 401 (comparação segura em termos de timing).

Transporte 2 – Unix-Domain-Socket: AF_UNIX/SOCK_STREAM em /tmp/punchbuddy.sock, permissões de ficheiro 0600. Sem stack IP → invisível para filtros de rede (Defender). Protocolo de linhas, loop de accept com threads, fecho limpo via atexit + handlers de Quit/Signal + cleanup de socket obsoleto.

def command_dispatch(self, command, args=None):
    cmd = _COMMAND_ALIASES.get(command.lower(), command.lower())
    handler = self._command_table().get(cmd)
    if handler is None: return (False, f"unknown command: {command}")
    return handler(args)        # → (ok, msg)

Sistema de configurações Config

As configurações são guardadas como JSON em ~/.punchbuddy/settings.json. No carregamento é feito um deep-merge recursivo com os valores por defeito: chaves recém-adicionadas no código são acrescentadas automaticamente, os valores existentes do utilizador são preservados.

def _deep_merge(default, override):
    """Neue Keys aus default werden ergänzt; override gewinnt bei Konflikten."""
    if isinstance(default, dict) and isinstance(override, dict):
        out = dict(default)
        for k, v in override.items():
            out[k] = _deep_merge(default[k], v) if k in default else v
        return out
    return override

As entradas de preset recebem adicionalmente um merge de template (_PRESET_TEMPLATE), para que chaves de preset recém-adicionadas também apareçam em presets antigos guardados.
