PunchBuddy

Application d'automatisation pour le Punch-In, le transport, l'export et Avid Interplay de Pro Tools.
Application de barre de menus (macOS) avec intégration Stream Deck, commande Vocaster et normalisation de la sonie selon EBU R128.

v 1.3  ·  Créé par Jens Rühl  ·  06.2026

1 · Menus de réglages

Tous les réglages sont enregistrés sous ~/.punchbuddy/settings.json et rechargés automatiquement au démarrage suivant. La fenêtre s'ouvre via Barre de menus → Réglages. Les modifications ne prennent effet qu'après un clic sur Enregistrer.

Onglet : Pistes (Profil A & B)

PunchBuddy connaît deux profils de pistes indépendants. Le profil A est déclenché par le déclencheur principal, le profil B par le second déclencheur (menu ou déclencheur web /trigger2). Cela permet par exemple de basculer rapidement entre deux configurations d'enregistrement différentes – par exemple Locuteur 1 vs. Locuteur 2.

Section Stream Deck

PunchBuddy embarque directement le plugin Stream Deck et les lanceurs de touches (intégrés dans l'application). Trois boutons :

BoutonEffet
Installer le pluginOuvre le com.punchbuddy.control.streamDeckPlugin fourni – Stream Deck affiche sa propre boîte de dialogue d'installation. Ensuite, l'action « PunchBuddy Befehl » avec menu déroulant de commandes est disponible.
Enregistrer le plugin (Downloads)Copie le fichier du plugin dans ~/Downloads et l'affiche dans le Finder (pour le transmettre / l'installer manuellement).
Créer un lanceur de touchesConstruit pour chaque commande une minuscule .app dans ~/Applications/PunchBuddy Launchers, qui envoie la commande via nc -U au Unix-Socket configuré. Utilisable dans Stream Deck via l'action intégrée « System → Öffnen » – sans installation de plugin.
Pourquoi deux voies ? Le plugin (une action, commande via menu déroulant) est pratique, mais nécessite Stream Deck ≥ 6.5. Les lanceurs fonctionnent avec n'importe quelle version de Stream Deck, entièrement sans installation de plugin. Les deux atteignent le même socket local – totalement sans réseau.

Onglet : Déclencheur web

Toutes les fonctions déclenchables passent par une API de commande centrale (command_dispatch). Celle-ci est desservie par deux transports :

  • Serveur HTTP (ThreadingMixIn + TCPServer, port par défaut 8899) – pour Stream Deck, Companion ou n'importe quel client HTTP.
  • Unix-Domain-Socket (/tmp/punchbuddy.sock) – entièrement sans réseau (ni TCP ni loopback), donc par principe indétectable par des filtres réseau comme Microsoft Defender. Protégé par des droits de fichier 0600 (uniquement l'utilisateur connecté).

Onglet : Presets

Il existe 8 presets, chacun enregistrant un état de travail complet. Un preset peut être chargé manuellement via l'onglet ou par déclencheur web /preset/1 à /preset/8.

Chaque entrée de preset enregistre : pistes Record A/B, pistes Monitor A/B, pistes d'export, pistes Monitor du Play, pistes de sonie ainsi que tous les réglages Bool/String/Int/Float comme loudness_enabled, interplay_enabled, target_lufs etc.

Astuce : Les presets peuvent être chargés via Companion ou Stream Deck. Cela permet de basculer entièrement entre différents formats de production (journal télévisé, magazine, …) d'une seule pression de touche.

2 · Processus techniques

Démarrage

1
Définir le nom du bundle et du processus Avant l'import de rumps/AppKit, CFBundleName = 'PunchBuddy' est écrit dans l'InfoDictionary et NSProcessInfo.setProcessName_ est appelé, afin que « Python » n'apparaisse pas dans le Dock et le Moniteur d'activité.
2
Initialiser le logging & le diagnostic de crash punchbuddy/log.py crée ~/.punchbuddy/PunchBuddy.log, ouvre PunchBuddy_crash.log pour faulthandler (SIGSEGV/SIGABRT) et installe threading.excepthook / sys.excepthook.
3
Charger les réglages & migration ~/.punchbuddy/settings.json est combiné par un Deep-Merge avec les valeurs par défaut. Si un ancien dossier ~/.autopunchin/ existe, il est migré une seule fois.
4
La boucle d'application rumps démarre L'application de barre de menus (PunchBuddyApp) est instanciée. Le serveur HTTP, le KeepAlive PTSL et le timer Vocaster sont démarrés en tant que threads/timers daemon.
5
Vérification Pre-Roll au démarrage (thread d'arrière-plan, délai de 2 s) Le Pre-Roll est mis sur ARRÊT (PTSL), le cache de pistes est rempli. Les deux se produisent en arrière-plan, afin que le démarrage ne bloque pas.
6
Initialisation Vocaster Détection de l'appareil USB via libusb. Si un Vocaster est trouvé : 48V activé (optionnel) + application du routing enregistré (optionnel) + ajout des entrées de menu.

Punch-In (Record) Record

Déclenché via le menu, Stream Deck (/trigger ou /trigger2) ou déclencheur web. S'exécute entièrement dans un thread daemon, afin que l'interface reste réactive.

1
Garde : déjà en cours ? state.running_lock empêche les enregistrements parallèles. Si un enregistrement est déjà en cours, le déclencheur est ignoré.
2
Interroger le cache de pistes (PTSL) Noms de pistes issus du cache de session (chargés une seule fois au démarrage). Si vide : appel PTSL frais track_list(). Les pistes de la liste des réglages sont mises en correspondance avec le cache.
3
Pre-Roll ACTIVÉ (PTSL, 3 tentatives) ensure_preroll_on() lit l'état actuel du Pre-Roll via GetTimelineSelection, puis le définit avec SetTimelineSelection(pre_roll_enabled=TB_True). Solution de repli sur Cmd+K (CGEvent) si PTSL échoue.
4
Record-Enable (PTSL, 3 tentatives, timeout de 15 s chacune) set_track_record_enable_state(tracks, True). Timeout de 15 s en raison d'une charge d'écriture NEXIS possible. En cas d'erreur : avertissement dans le journal, le processus se poursuit malgré tout.
5
Transport-Arm + Play (PTSL) On vérifie d'abord si le transport est déjà armé (transport_armed()). Si non : toggle_record_enable() → toggle_play_state(). Pro Tools démarre l'enregistrement.
6
Attendre que le transport démarre (max 5 s) Polling sur transport_state() toutes les 50 ms jusqu'à TS_TransportRecording ou TS_TransportPlaying.
7
Attendre que le transport s'arrête (max 10 minutes) Polling toutes les 300 ms. En cas de TS_TransportIsStopping (PT écrit sur NEXIS), on bascule sur un intervalle de 3 s afin de réduire la charge PTSL. Après 10 timeouts consécutifs : réinitialisation du moteur (Engine-Reset).
8
Transport-Arm DÉSACTIVÉ + Pre-Roll DÉSACTIVÉ Le Transport-Arm n'est désactivé que s'il a été activé par PunchBuddy. Le Pre-Roll est réinitialisé avec restore_preroll().

Play & Play Custom Play

Play (Input Monitor) lit l'état actuel du transport. Si PT est en marche → chemin Stop. Si PT est arrêté → chemin Start : activer l'Input Monitor des pistes configurées (ou sélectionnées), puis démarrer la lecture. À l'arrêt, les pistes Monitor sont de nouveau désactivées.

Play Custom fonctionne comme Play, mais définit avant le démarrage des états Mute prédéfinis sur deux pistes configurables (typiquement : couper KH2, laisser le monitoring studio actif). À l'arrêt, ces Mutes sont de nouveau réinitialisés.

Stop Transport

La commande Stop est explicitement découplée de state.running – elle fonctionne aussi pendant un enregistrement en cours. Elle utilise son propre _stop_lock (empêche un double Stop, mais ne bloque pas un Punch-In parallèle). toggle_play_state() correspond à la barre d'espace dans PT.

Déplacer l'audio (déplacement de piste) Edit

Déplace l'intégralité du matériel des pistes source vers les pistes cible – à une position temporelle identique – entièrement via PTSL cut/paste. Déclenché via le menu « Déplacer l'audio » ou le déclencheur web /move_audio (run_move_audio() dans transport.py).

1
Activer le contexte de sélection d'édition Via _protools_selection_context : sélectionner l'outil Selector et activer « Link Timeline and Edit Selection » + « Link Track and Edit Selection ». Ce n'est qu'ainsi que select_tracks_by_name / set_timeline_selection pilotent l'Edit-Selection sur laquelle cut/paste travaillent. Les réglages d'origine sont ensuite restaurés.
2
Déplacer par paires, piste par piste (src[i] → tgt[i]) L'affectation se fait explicitement par index, et non via le mapping top→bottom de PT. Chaque piste est déplacée individuellement ; avant chaque piste source, la sélection est mise à une longueur nulle, afin que les pistes vides (sélection périmée) soient correctement ignorées.
3
Couper par clip (PTSL) select_all_clips_on_track(src[i]) crée une Edit-Selection au clip près, puis cut(). Le début de clip d'origine est mémorisé. Des délais de stabilisation entre les étapes, car PT doit traiter chaque changement de sélection.
4
Coller sur la piste cible à la même position (PTSL) select_tracks_by_name([tgt[i]]) + curseur sur le début de clip mémorisé (set_timeline_selection(start, start)), puis paste(). Chaque piste atterrit à sa propre position d'origine – indépendamment du timecode de début.
Deux enseignements du test en conditions réelles : (1) PTSL cut/paste opèrent sur l'Edit-Selection – sans les options Link, cut ne coupe qu'une seule piste et paste atterrit au mauvais endroit. (2) Une sélection multi-pistes basée sur une plage temporelle coupe de manière peu fiable un matériel qui ne commence pas exactement sur une frame (sous-frames, p. ex. 10:00:00:11.64). C'est pourquoi on procède par clip et piste par piste. Vérifié contre Pro Tools : tant le matériel exactement à la frame (A1/A2 → A3/A4) que le matériel décalé en sous-frames (A5/A6 → A7/A8) est déplacé à la position exacte, et un aller-retour rétablit l'état d'origine.

Workflow d'export complet Export

Démarré via le menu ou un déclencheur web. Comprend, selon la configuration : arrêt du transport → Consolidate → WAV → Loudness → AAF → export Interplay. Toute la gestion des pistes et des sélections passe par PTSL – aucune frappe clavier simulée n'est utilisée pour cela.

1
Arrêter le transport s'il est actif (PTSL) nouveau 06.2026 _ensure_transport_stopped() lit transport_state(). Si PT est encore en marche (Record/Play), toggle_play_state() (Stop) est envoyé, on attend un arrêt confirmé et on laisse 2 s à PT pour finaliser le fichier. Si le transport est déjà arrêté, l'export se poursuit immédiatement. Protège contre les dommages de session lorsque l'utilisateur déclenche l'export sans arrêt préalable. S'exécute dans les cinq fonctions d'export.
2
Détecter automatiquement la piste vidéo (PTSL) _detect_video_track() : recherche la piste avec type=4 dans le cache de pistes PTSL. Solution de repli sur le nom saisi dans les réglages.
3
Afficher les pistes + déterminer la fin de la vidéo (PTSL) set_track_hidden_state(tracks, False) affiche les pistes d'export masquées. select_all_clips_on_track(video_track) + get_timeline_selection() fournissent l'Out-Time (fin de la vidéo).
4
Sélectionner les pistes & étendre la sélection (PTSL) select_tracks_by_name(export_tracks) marque les pistes d'export, set_timeline_selection(in_time, out_time) définit la plage temporelle, extend_selection_to_target_tracks(export_tracks) étend l'Edit-Selection à toutes les pistes cibles. Le tout via PTSL – aucune simulation de clavier.
5
Consolidate (PTSL) + attendre l'achèvement engine.consolidate_clip() déclenche le Consolidate via PTSL. L'attente de l'achèvement utilise AppleScript : _wait_for_consolidate_window_gone() vérifie les titres de fenêtres de PT (correspondance de début de mot sur « Consolidating »/« Zusammenführen »). Le timeout est calculé à partir de la longueur de la sélection.
6
Attendre la stabilité des fichiers WAV _wait_for_consolidated_files() : vérifie toutes les 100 ms si les fichiers WAV dans le dossier Audio Files/ sont stables (taille inchangée pendant 500 ms). Les pistes sans matériel ne bloquent pas.
7
Normalisation de la sonie (optionnel) Pour chaque piste dans loudness_tracks : lire le WAV (soundfile), mesurer les LUFS (pyloudnorm), calculer la correction de gain, appliquer un limiteur True-Peak, écraser le fichier. PT est ensuite invité à relire.
8
Export AAF (optionnel) Via automatisation UI AppleScript dans la boîte de dialogue d'export de PT. Embedded ou Reference selon la configuration.
9
Export Interplay (optionnel) Voir la section dédiée ci-dessous.

Export Interplay Interplay

Entièrement via automatisation UI (AppleScript + CGEvent sur les processus Pro Tools et interplayAccess). PTSL n'offre pas d'API pour cela.

1
Activer PT, ouvrir la boîte de dialogue d'export AppleScript active Pro Tools, ouvre Clip → Export to MediaCentral. PT affiche une boîte de dialogue modale avec barre de progression.
2
Surveiller le journal PT via kqueue Le journal PT (~/Library/Logs/Avid/Pro_Tools_*.txt) est observé en direct via kqueue (événement VNODE). Chaque nouvelle ligne est vérifiée pour SendDialogOnScreen … error msg = …. « successfully » → succès ; autre texte → erreur.
3
Confirmer la boîte de dialogue de succès PT AppleScript envoie Return sur la boîte de dialogue OK de PT. _wait_for_pt_modal_dismissed() attend via kqueue jusqu'à ce que PT écrive l'événement de fermeture de la boîte de dialogue dans le journal.
4
Activer Interplay Access + naviguer dans le workspace AppleScript active interplayAccess. Des pressions de la touche Tab naviguent dans l'arborescence du workspace selon le nombre d'étapes configuré.
5
Renommer (optionnel) F2 ouvre le champ de renommage. Le nom est lu (Cmd+A → Cmd+C), manipulé selon les paramètres (Trim + Préfixe + Suffixe), réécrit et confirmé par Entrée. Le champ reste alors ouvert – pas de double ouverture.

Import Interplay Interplay

Ouvre une séquence dans Interplay Access et la transmet à Pro Tools. En option, la session PT actuelle est automatiquement fermée au préalable (import_close_session). PT confirme l'import par une fenêtre de dialogue, qui est lue via AppleScript.

Normalisation de la sonie (EBU R128) Loudness

S'exécute après le Consolidate, avant l'export AAF. Toutes les étapes se déroulent en mémoire vive – le fichier WAV n'est écrasé qu'à la fin.

  1. Trouver le <Spurname>*.wav le plus récent dans le dossier Audio Files/ (par mtime)
  2. Charger l'audio avec soundfile.read()
  3. Mesurer la sonie intégrée avec pyloudnorm.Meter.integrated_loudness()
  4. Calculer la différence de gain : gain_db = target_lufs − current_lufs
  5. Appliquer le gain : data *= 10^(gain_db/20)
  6. Limiteur True-Peak : les crêtes au-dessus de max_truepeak sont ramenées
  7. Écraser le fichier avec soundfile.write()
  8. Pro Tools : Clip → Refresh via AppleScript afin que PT lise le nouveau fichier

Modèle de threading

PunchBuddy est une application à Main-Thread unique (rumps/AppKit) qui délègue toutes les opérations de longue durée à des threads daemon.

Règle critique : Tous les appels AppKit/NSStatusItem doivent avoir lieu dans le Main-Thread. _dispatch_main(fn) dans uikit.py utilise AppHelper.callAfter() (performSelectorOnMainThread:), qui réveille la boucle d'exécution de manière thread-safe. NSRunLoop.performBlock_() n'est pas thread-safe et ne doit pas être utilisé.

PTSL / gRPC Core

PTSL (Pro Tools Scripting Layer) est une API basée sur gRPC d'Avid, qui écoute sur le port 31416. Le package Python ptsl wraps les définitions Proto. PunchBuddy gère une instance singleton (_engine_instance) avec reconnexion automatique.

Wrapper de deadline gRPC (engine.py:_install_grpc_deadline)

Le package ptsl ne transmet aucun timeout au stub gRPC. PunchBuddy patche donc raw_client.SendGrpcRequest avec un wrapper qui définit timeout=_current_grpc_deadline(). La valeur provient du Thread-Local-Storage (_grpc_deadline_tls) et est définie par _ptsl_call() avant chaque appel.

_ptsl_call() – appel sérialisé

def _ptsl_call(fn, *args, label="", timeout=15.0):
    # Verrou exclusif : empêche les appels gRPC parallèles (protection contre le délai NEXIS)
    if not _ptsl_lock.acquire(timeout=6.0):
        return False, None

    _grpc_deadline_tls.value = timeout  # Deadline pour cet appel
    try:
        return True, fn(*args)
    except grpc.RpcError as e:
        if e.code() == grpc.StatusCode.DEADLINE_EXCEEDED:
            ...  # Timeout → abandonner le moteur
        _reset_engine()
        return False, e
    finally:
        _ptsl_lock.release()

Cache de pistes

Les noms de pistes ne changent pas pendant une session. _get_cached_track_names() les charge une seule fois (au démarrage ou au premier déclencheur Record) et les conserve en mémoire. Le cache est vidé lors d'un Engine-Reset / changement de session via _invalidate_session_tracks(). run_play() continue de lire les pistes en direct, car la sélection actuelle est nécessaire en temps réel.

AppKit & Main-Thread-Dispatch AppKit

# uikit.py – dispatch thread-safe sur le Main-Thread
def _dispatch_main(fn):
    def _safe():
        try:
            fn()
        except Exception:
            logging.exception("_dispatch_main: Fehler im Main-Thread-Block")
    try:
        from PyObjCTools import AppHelper
        AppHelper.callAfter(_safe)   # = performSelectorOnMainThread:
    except Exception:
        pass

Pourquoi pas NSRunLoop.performBlock_() ? – Cette méthode n'est pas thread-safe et ne réveille pas la boucle d'exécution. Les appels depuis des threads de travail peuvent se perdre silencieusement ou provoquer des situations de concurrence (race conditions). AppHelper.callAfter() utilise en interne performSelectorOnMainThread:withObject:waitUntilDone: (wait=False) et constitue le pattern correct.

CGEvent – Commande du clavier Input

Pour les rares actions sans API PTSL, les frappes de touches sont envoyées sous forme de CGEvents directement au PID de Pro Tools – cela fonctionne même lorsque PT n'est pas au premier plan. En pratique, il ne reste de cela que le fallback Pre-Roll (Cmd+K), au cas où la définition via PTSL SetTimelineSelection échoue trois fois. L'extension de sélection lors de l'export ne passe pas par le clavier, mais par PTSL extend_selection_to_target_tracks() ; l'ancienne fonction auxiliaire Shift+Ö (_send_shift_oe) n'est plus active dans le chemin de code actuel.

# keys.py
def _send_key_to_app(pid, vk, cmd=False, shift=False):
    src = Quartz.CGEventSourceCreate(kCGEventSourceStateHIDSystemState)
    ev_dn = Quartz.CGEventCreateKeyboardEvent(src, vk, True)
    if cmd:
        Quartz.CGEventSetFlags(ev_dn, kCGEventFlagMaskCommand)
    Quartz.CGEventPostToPid(pid, ev_dn)
    Quartz.CGEventPostToPid(pid, Quartz.CGEventCreateKeyboardEvent(src, vk, False))

Constantes VK (codes matériels macOS, indépendants de la disposition) : _VK_K=40 (Cmd+K = fallback Pre-Roll), _VK_F2=120 (Interplay Rename), _VK_ESC=53, _VK_RETURN=36. Les CGEvents sont en outre utilisés dans l'automatisation UI d'Interplay (F2/Tab/Cmd+A/C/V), car Interplay Access n'a pas d'API PTSL.

AppleScript / Automatisation UI UI

AppleScript via osascript (subprocess) est utilisé pour les actions qui n'ont pas d'API PTSL :

  • Lire les titres de fenêtres de PT (name of every window) – p. ex. pour attendre la fin de l'opération Consolidate
  • Déclencher le menu d'export de PT (Clip → Export to MediaCentral). Le Consolidate lui-même s'exécute via PTSL consolidate_clip(), pas via AppleScript.
  • Piloter Interplay Access (navigation par Tab, renommage F2, Cmd+C/V)
  • Lire le texte statique des fenêtres de dialogue de PT (résultat de l'export)
  • Activer une application (tell application "interplayAccess" to activate)

Tous les appels AppleScript sont encapsulés via _run_applescript(script, timeout=30), qui lance un processus subprocess.run(). En cas de timeout, la fonction renvoie un code de retour d'erreur.

Autorisation Accessibility : L'accès AppleScript à System Events nécessite que PunchBuddy (ou Python) soit autorisé sous Réglages Système → Confidentialité → Accessibilité.

kqueue – Surveillance de fichiers macOS

kqueue est le mécanisme de notification d'événements du noyau macOS. PunchBuddy l'utilise pour surveiller le fichier journal de PT, sans polling :

# export.py – Simplifié
fd = os.open(log_path, os.O_RDONLY | os.O_NONBLOCK)
kq = select.kqueue()
ke = select.kevent(fd,
    filter=select.KQ_FILTER_VNODE,
    flags=select.KQ_EV_ADD | select.KQ_EV_ENABLE | select.KQ_EV_CLEAR,
    fflags=select.KQ_NOTE_WRITE | select.KQ_NOTE_EXTEND)

while time.time() < deadline:
    events = kq.control([ke], 4, min(remaining, 2.0))  # bloque jusqu'à un événement
    if events:
        # Lire les nouveaux octets, vérifier le marqueur
        ...

Avantages par rapport au polling : réaction immédiate lors d'un événement de fichier (pas d'intervalle fixe de 0,3 s), aucune charge CPU entre les écritures. Solution de repli sur un polling de 0,3 s si kqueue n'est pas disponible pour une quelconque raison.

API de commande : Dispatcher + transports API

Au cœur se trouve une seule table de commandes command_dispatch(command, args) → (ok, msg) (_command_table()). UNE entrée par fonction (record_a/b, play, play_custom, goto_start, move_audio, import, tous les exports, preset, vocaster_*, ping/status/list). Tous les transports appellent exclusivement ce dispatcher – ainsi HTTP et Socket partagent automatiquement le debounce et la sérialisation des exports.

Transport 1 – HTTP : ThreadingMixIn + TCPServer intégré. Pour chaque requête de déclenchement, immédiatement 200 OK, la fonction se poursuit en file d'attente. Timeout de socket (30 s) contre les clients lents. En cas de bind sur 0.0.0.0 (LAN), un token (secrets.token_urlsafe(16)) est généré automatiquement ; les requêtes sans token correct reçoivent 401 (comparaison à durée constante).

Transport 2 – Unix-Domain-Socket : AF_UNIX/SOCK_STREAM sur /tmp/punchbuddy.sock, droits de fichier 0600. Aucune pile IP → invisible pour les filtres réseau (Defender). Protocole en ligne, boucle d'acceptation threadée, fermeture propre via atexit + gestionnaires Quit/Signal + nettoyage des sockets périmés.

def command_dispatch(self, command, args=None):
    cmd = _COMMAND_ALIASES.get(command.lower(), command.lower())
    handler = self._command_table().get(cmd)
    if handler is None: return (False, f"unknown command: {command}")
    return handler(args)        # → (ok, msg)

Système de réglages Config

Les réglages sont enregistrés en JSON sous ~/.punchbuddy/settings.json. Au chargement, un Deep-Merge récursif est effectué avec les valeurs par défaut : les nouvelles clés ajoutées par le code sont automatiquement complétées, les valeurs utilisateur existantes sont conservées.

def _deep_merge(default, override):
    """Les nouvelles clés de default sont ajoutées ; override l'emporte en cas de conflit."""
    if isinstance(default, dict) and isinstance(override, dict):
        out = dict(default)
        for k, v in override.items():
            out[k] = _deep_merge(default[k], v) if k in default else v
        return out
    return override

Les entrées de preset reçoivent en plus un merge de template (_PRESET_TEMPLATE), afin que les clés de preset nouvellement ajoutées apparaissent aussi dans les anciens presets enregistrés.
