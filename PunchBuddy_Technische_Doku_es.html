PunchBuddy

App de automatización para Pro Tools Punch-In, transporte, exportación y Avid Interplay.
App de barra de menús (macOS) con integración de Stream Deck, control de Vocaster y normalización de sonoridad según EBU R128.

v 1.3  ·  Creado por Jens Rühl  ·  06.2026

1 · Menús de ajustes

Todos los ajustes se guardan en ~/.punchbuddy/settings.json y se cargan automáticamente en el siguiente arranque. La ventana se abre a través de Barra de menús → Ajustes. Los cambios solo se aplican tras hacer clic en Guardar.

Pestaña: Pistas (Perfil A & B)

PunchBuddy conoce dos perfiles de pistas independientes. El Perfil A se activa mediante el disparador principal, y el Perfil B mediante el segundo disparador (menú o webtrigger /trigger2). Esto permite, por ejemplo, alternar rápidamente entre dos configuraciones de grabación distintas, como Locutor 1 vs. Locutor 2.

Sección de Stream Deck

PunchBuddy trae consigo directamente el plugin de Stream Deck y los lanzadores de botones (empaquetados dentro de la app). Tres botones:

BotónEfecto
Instalar pluginAbre el com.punchbuddy.control.streamDeckPlugin incluido; Stream Deck muestra su propio diálogo de instalación. Después queda disponible la acción «Comando de PunchBuddy» con un desplegable de comandos.
Guardar plugin (Descargas)Copia el archivo del plugin a ~/Downloads y lo muestra en el Finder (para compartir/instalar manualmente).
Crear lanzadores de botonesGenera, por cada comando, una minúscula .app en ~/Applications/PunchBuddy Launchers que envía el comando mediante nc -U al Unix-Socket configurado. Se puede usar en Stream Deck con la acción integrada «Sistema → Abrir», sin instalar el plugin.
¿Por qué dos vías? El plugin (una acción, comando por desplegable) es cómodo, pero requiere Stream Deck ≥ 6.5. Los lanzadores funcionan con cualquier versión de Stream Deck sin instalar plugin alguno. Ambos apuntan al mismo Socket local, completamente sin red.

Pestaña: Webtrigger

Todas las funciones activables pasan por una API de comandos central (command_dispatch). Esta se atiende mediante dos transportes:

  • Servidor HTTP (ThreadingMixIn + TCPServer, puerto predeterminado 8899): para Stream Deck, Companion o cualquier cliente HTTP.
  • Unix-Domain-Socket (/tmp/punchbuddy.sock): completamente sin red (sin TCP/Loopback) y, por tanto, en principio no detectable por filtros de red como Microsoft Defender. Protegido mediante permisos de archivo 0600 (solo el usuario que ha iniciado sesión).
CampoClave de ajusteDescripción
Puertohttp_portPuerto TCP del servidor HTTP (predeterminado: 8899). Los cambios solo se aplican tras reiniciar. Si el puerto está ocupado, se elige automáticamente un puerto libre.
Bind-Hosthttp_bind_host127.0.0.1 = solo local (seguro, no se necesita token). Cualquier otra IP = accesible en la red → se genera un token automáticamente.
Tokenwebtrigger_tokenToken de autenticación opcional (solo HTTP). En un bind de red se genera automáticamente. Debe enviarse como ?token=… o como cabecera X-Auth-Token.
Unix-Socket activounix_socket_enabledActiva/desactiva el Socket sin red (predeterminado: activado).
Ruta del Socketunix_socket_pathRuta del Unix-Socket (predeterminado: /tmp/punchbuddy.sock). Protocolo: una línea <befehl> [arg], respuesta OK … / ERR …. Prueba: printf 'play' | nc -U /tmp/punchbuddy.sock.

Pestaña: Presets

Existen 8 presets, cada uno de los cuales guarda un estado de trabajo completo. Un preset puede cargarse manualmente desde la pestaña o mediante webtrigger /preset/1 hasta /preset/8.

Cada entrada de preset guarda: pistas de grabación A/B, pistas de monitor A/B, pistas de exportación, pistas de monitor en Play, pistas de sonoridad, así como todos los ajustes booleanos/de texto/enteros/decimales como loudness_enabled, interplay_enabled, target_lufs, etc.

Consejo: Los presets pueden cargarse mediante Companion o Stream Deck. Así se pueden alternar por completo distintos formatos de producción (informativo, revista, …) con una sola pulsación.

2 · Procesos técnicos

Arranque

1
Definir el nombre del bundle y del proceso Antes de importar rumps/AppKit se escribe CFBundleName = 'PunchBuddy' en el InfoDictionary y se llama a NSProcessInfo.setProcessName_, para que en el Dock y el Monitor de Actividad no aparezca «Python».
2
Inicializar el registro (logging) y el diagnóstico de fallos punchbuddy/log.py crea ~/.punchbuddy/PunchBuddy.log, abre PunchBuddy_crash.log para faulthandler (SIGSEGV/SIGABRT) e instala threading.excepthook / sys.excepthook.
3
Cargar ajustes y migración ~/.punchbuddy/settings.json se combina mediante Deep-Merge con los valores predeterminados. Si existe una carpeta antigua ~/.autopunchin/, se migra una única vez.
4
Arranca el bucle de la app rumps Se instancia la app de barra de menús (PunchBuddyApp). El servidor HTTP, el KeepAlive de PTSL y el temporizador de Vocaster se inician como threads/timers daemon.
5
Comprobación de Pre-Roll al arrancar (thread en segundo plano, 2s de retardo) El Pre-Roll se pone en OFF (PTSL) y se llena la caché de pistas. Ambas cosas ocurren en segundo plano para que el arranque no se bloquee.
6
Inicialización de Vocaster Detección de dispositivos USB vía libusb. Si se encuentra un Vocaster: 48V activado (opcional) + aplicar el routing guardado (opcional) + añadir entradas de menú.

Punch-In (grabación) Record

Se activa mediante menú, Stream Deck (/trigger o /trigger2) o webtrigger. Se ejecuta por completo en un thread daemon para que la interfaz siga respondiendo.

1
Guard: ¿ya está en marcha? state.running_lock impide grabaciones en paralelo. Si ya hay una en curso, el disparador se ignora.
2
Consultar la caché de pistas (PTSL) Nombres de pista de la caché de sesión (cargada una única vez al arrancar). Si está vacía: nueva llamada PTSL track_list(). Las pistas de la lista de ajustes se cotejan contra la caché.
3
Pre-Roll ON (PTSL, 3 intentos) ensure_preroll_on() lee el estado actual del Pre-Roll vía GetTimelineSelection y lo fija con SetTimelineSelection(pre_roll_enabled=TB_True). Alternativa a Cmd+K (CGEvent) si PTSL falla.
4
Record-Enable (PTSL, 3 intentos, 15s de timeout cada uno) set_track_record_enable_state(tracks, True). Timeout de 15s por la posible carga de escritura de NEXIS. En caso de error: aviso en el registro; el proceso continúa igualmente.
5
Armar transporte + Play (PTSL) Primero se comprueba si el transporte ya está armado (transport_armed()). Si no: toggle_record_enable() → toggle_play_state(). Pro Tools inicia la grabación.
6
Esperar hasta que el transporte esté en marcha (máx. 5s) Sondeo de transport_state() cada 50ms hasta TS_TransportRecording o TS_TransportPlaying.
7
Esperar hasta que el transporte se detenga (máx. 10 minutos) Sondeo cada 300ms. En TS_TransportIsStopping (PT escribe en NEXIS) se conmuta a un intervalo de 3s para reducir la carga de PTSL. Tras 10 timeouts consecutivos: reinicio del motor.
8
Desarmar transporte + Pre-Roll OFF El armado del transporte solo se desactiva si lo activó PunchBuddy. El Pre-Roll se restablece con restore_preroll().

Play & Play Custom Play

Play (monitor de entrada) lee el estado actual del transporte. Si PT está en marcha → ruta de Stop. Si PT está detenido → ruta de inicio: activar el monitor de entrada de las pistas configuradas (o seleccionadas) y luego iniciar la reproducción. Al detener, las pistas de monitor se vuelven a desactivar.

Play Custom funciona como Play, pero antes de iniciar establece estados de mute definidos en dos pistas configurables (típico: silenciar KH2, dejar activa la escucha de estudio). Al detener, estos mutes se restablecen.

Stop Transport

El comando Stop está desacoplado explícitamente de state.running: funciona también durante una grabación en curso. Utiliza su propio _stop_lock (impide un doble Stop, pero no bloquea un Punch-In en paralelo). toggle_play_state() equivale a la barra espaciadora en PT.

Mover audio (Track Move) Edit

Mueve todo el material de las pistas de origen a las pistas de destino —en la misma posición temporal— completamente vía PTSL cut/paste. Se activa mediante el menú «Mover audio» o el webtrigger /move_audio (run_move_audio() en transport.py).

1
Activar el contexto de Edit-Selection Mediante _protools_selection_context: seleccionar la herramienta Selector y activar «Link Timeline and Edit Selection» + «Link Track and Edit Selection». Solo así select_tracks_by_name / set_timeline_selection controlan la Edit-Selection sobre la que operan cut/paste. Los ajustes originales se restauran después.
2
Mover por parejas, pista a pista (src[i] → tgt[i]) La asignación se hace explícitamente por índice, no mediante el mapeo top→bottom de PT. Cada pista se mueve por separado; antes de cada pista de origen la selección se pone a longitud cero, para que las pistas vacías (selección obsoleta) se omitan correctamente.
3
Cortar basándose en clips (PTSL) select_all_clips_on_track(src[i]) genera una Edit-Selection precisa a nivel de clip, y luego cut(). Se memoriza el inicio original del clip. Hay retardos de asentamiento (settling) entre los pasos, ya que PT debe procesar cada cambio de selección.
4
Pegar en la pista de destino en la misma posición (PTSL) select_tracks_by_name([tgt[i]]) + cursor en el inicio de clip memorizado (set_timeline_selection(start, start)), y luego paste(). Cada pista aterriza en su propia posición original, independiente del timecode de inicio.
Dos conclusiones de la prueba en vivo: (1) PTSL cut/paste operan sobre la Edit-Selection; sin las opciones de Link, cut corta solo una pista y paste aterriza mal. (2) Una selección multipista basada en rango temporal corta de forma poco fiable cuando el material no empieza con precisión de frame (subframes, p. ej. 10:00:00:11.64). Por eso se hace basándose en clips y pista a pista. Verificado contra Pro Tools: tanto el material con precisión de frame (A1/A2 → A3/A4) como el desplazado por subframes (A5/A6 → A7/A8) se mueve a la posición exacta, y el round-trip restaura el estado original.

Flujo de exportación completo Export

Se inicia mediante menú o webtrigger. Comprende, según la configuración: Stop del transporte → Consolidate → WAV → Loudness → AAF → exportación a Interplay. Todo el control de pistas y selección pasa por PTSL; no se usan pulsaciones de tecla simuladas para ello.

1
Detener el transporte si está activo (PTSL) nuevo 06.2026 _ensure_transport_stopped() lee transport_state(). Si PT sigue en marcha (Record/Play), se envía toggle_play_state() (Stop), se espera a la detención confirmada y se dan 2 s a PT para finalizar los archivos. Si el transporte ya está detenido, la exportación continúa de inmediato. Protege frente a daños en la sesión cuando el usuario activa la exportación sin detener antes. Se ejecuta en las cinco funciones de exportación.
2
Autodetectar la pista de vídeo (PTSL) _detect_video_track(): busca la pista con type=4 en la caché de pistas de PTSL. Alternativa: el nombre indicado en los ajustes.
3
Mostrar pistas + determinar el final del vídeo (PTSL) set_track_hidden_state(tracks, False) muestra las pistas de exportación ocultas. select_all_clips_on_track(video_track) + get_timeline_selection() proporcionan el Out-Time (final del vídeo).
4
Seleccionar pistas y ampliar la selección (PTSL) select_tracks_by_name(export_tracks) marca las pistas de exportación, set_timeline_selection(in_time, out_time) fija el rango temporal, y extend_selection_to_target_tracks(export_tracks) amplía la Edit-Selection a todas las pistas de destino. Todo vía PTSL, sin simulación de teclado.
5
Consolidate (PTSL) + esperar a que finalice engine.consolidate_clip() lanza el Consolidate vía PTSL. La espera a que finalice usa AppleScript: _wait_for_consolidate_window_gone() comprueba los títulos de ventana de PT (coincidencia por inicio de palabra con «Consolidating»/«Zusammenführen»). El timeout se calcula a partir de la longitud de la selección.
6
Esperar a la estabilidad del archivo WAV _wait_for_consolidated_files(): comprueba cada 100ms si los archivos WAV de la carpeta Audio Files/ son estables (tamaño sin cambios durante 500ms). Las pistas sin material no bloquean.
7
Normalización de sonoridad (opcional) Para cada pista en loudness_tracks: leer el WAV (soundfile), medir LUFS (pyloudnorm), calcular la corrección de ganancia, aplicar el limitador de True-Peak, sobrescribir el archivo. Después se solicita a PT que lo relea.
8
Exportación AAF (opcional) Vía automatización de UI con AppleScript en el diálogo de exportación de PT. Embedded o Reference según la configuración.
9
Exportación a Interplay (opcional) Véase la sección propia más abajo.

Exportación a Interplay Interplay

Completamente vía automatización de UI (AppleScript + CGEvent sobre los procesos Pro Tools e interplayAccess). PTSL no ofrece ninguna API para esto.

1
Activar PT, abrir el diálogo de exportación AppleScript activa Pro Tools y abre Clip → Export to MediaCentral. PT muestra un diálogo modal con barra de progreso.
2
Monitorizar el log de PT vía kqueue El log de PT (~/Library/Logs/Avid/Pro_Tools_*.txt) se observa en vivo mediante kqueue (evento VNODE). Cada línea nueva se comprueba en busca de SendDialogOnScreen … error msg = …. «successfully» → éxito; otro texto → error.
3
Confirmar el diálogo de éxito de PT AppleScript envía Return al diálogo OK de PT. _wait_for_pt_modal_dismissed() espera vía kqueue hasta que PT escribe en el log el evento de cierre del diálogo.
4
Activar Interplay Access + navegar por el workspace AppleScript activa interplayAccess. Las pulsaciones de Tab navegan por el árbol de workspaces hasta el número de pasos configurado.
5
Renombrar (opcional) F2 abre el campo de renombrado. El nombre se lee (Cmd+A → Cmd+C), se manipula según lo configurado (Trim + Prefijo + Sufijo), se vuelve a escribir y se confirma con Enter. El campo permanece abierto durante el proceso, sin abrirse por partida doble.

Importación desde Interplay Interplay

Abre una secuencia en Interplay Access y la entrega a Pro Tools. Opcionalmente, la sesión actual de PT se cierra antes de forma automática (import_close_session). PT confirma la importación mediante un cuadro de diálogo que se lee vía AppleScript.

Normalización de sonoridad (EBU R128) Loudness

Se ejecuta tras el Consolidate, antes de la exportación AAF. Todos los pasos se realizan en memoria; el archivo WAV solo se sobrescribe al final.

  1. Encontrar el <Spurname>*.wav más reciente en la carpeta Audio Files/ (por mtime)
  2. Cargar el audio con soundfile.read()
  3. Medir la sonoridad integrada con pyloudnorm.Meter.integrated_loudness()
  4. Calcular la diferencia de ganancia: gain_db = target_lufs − current_lufs
  5. Aplicar la ganancia: data *= 10^(gain_db/20)
  6. Limitador de True-Peak: los picos por encima de max_truepeak se recalculan a la baja
  7. Sobrescribir el archivo con soundfile.write()
  8. Pro Tools: Clip → Refresh vía AppleScript para que PT lea el archivo nuevo

Control USB de Vocaster Vocaster

PunchBuddy se comunica directamente por USB (HID) con el Focusrite Vocaster. La clase vocaster_control.VocasterUSB encapsula las transacciones USB. Toda la lógica se ejecuta en el Main-Thread vía rumps.Timer (cada 400ms), porque las ventanas de AppKit solo pueden manejarse ahí.

FunciónProceso

Modelo de threading

PunchBuddy es una app de un único Main-Thread (rumps/AppKit) que delega todas las operaciones de larga duración a threads daemon.

Thread / TimerArchivoTarea

PTSLKeepAliveauto_punch_in.pyThread daemon; hace ping a transport_state() cada 120s para mantener caliente la conexión gRPC.
Threads de Punch-In / Play / Exporttransport.py / export.pyThreads daemon, iniciados con cada disparador. Se ejecutan hasta finalizar.

Regla crítica: Todas las llamadas a AppKit/NSStatusItem deben realizarse en el Main-Thread. _dispatch_main(fn) en uikit.py usa AppHelper.callAfter() (performSelectorOnMainThread:), que despierta el Run-Loop de forma segura entre threads. NSRunLoop.performBlock_() no es seguro entre threads y no debe utilizarse.

PTSL / gRPC Core

PTSL (Pro Tools Scripting Layer) es una API basada en gRPC de Avid que escucha en el puerto 31416. El paquete de Python ptsl envuelve las definiciones Proto. PunchBuddy gestiona una instancia singleton (_engine_instance) con reconexión automática.

Wrapper de deadline de gRPC (engine.py:_install_grpc_deadline)

El paquete ptsl no pasa ningún timeout al stub de gRPC. Por eso PunchBuddy parchea raw_client.SendGrpcRequest con un wrapper que establece timeout=_current_grpc_deadline(). El valor proviene de Thread-Local-Storage (_grpc_deadline_tls) y lo fija _ptsl_call() antes de cada llamada.

_ptsl_call() – llamada serializada

def _ptsl_call(fn, *args, label="", timeout=15.0):
    # Lock exclusivo: impide llamadas gRPC en paralelo (protección frente al retardo de NEXIS)
    if not _ptsl_lock.acquire(timeout=6.0):
        return False, None

    _grpc_deadline_tls.value = timeout  # Deadline para esta llamada
    try:
        return True, fn(*args)
    except grpc.RpcError as e:
        if e.code() == grpc.StatusCode.DEADLINE_EXCEEDED:
            ...  # Timeout → descartar el motor
        _reset_engine()
        return False, e
    finally:
        _ptsl_lock.release()

Caché de pistas

Los nombres de pista no cambian durante una sesión. _get_cached_track_names() los carga una única vez (al arrancar o en el primer disparador de grabación) y los mantiene en memoria. La caché se vacía al reiniciar el motor / cambiar de sesión mediante _invalidate_session_tracks(). run_play() sigue leyendo las pistas en vivo, ya que la selección actual se necesita en tiempo real.

AppKit & Main-Thread-Dispatch AppKit

# uikit.py – Dispatch seguro entre threads al Main-Thread
def _dispatch_main(fn):
    def _safe():
        try:
            fn()
        except Exception:
            logging.exception("_dispatch_main: Fehler im Main-Thread-Block")
    try:
        from PyObjCTools import AppHelper
        AppHelper.callAfter(_safe)   # = performSelectorOnMainThread:
    except Exception:
        pass

¿Por qué no NSRunLoop.performBlock_()? Ese método no es seguro entre threads y no despierta el Run-Loop. Las llamadas desde threads de trabajo pueden perderse silenciosamente o provocar condiciones de carrera. AppHelper.callAfter() usa internamente performSelectorOnMainThread:withObject:waitUntilDone: (wait=False) y es el patrón correcto.

CGEvent – control de teclado Input

Para las pocas acciones sin API de PTSL, las pulsaciones de tecla se envían como CGEvents directamente al PID de Pro Tools; esto funciona incluso cuando PT no está en primer plano. En la práctica, de esto solo queda la alternativa de Pre-Roll (Cmd+K), por si el ajuste vía PTSL SetTimelineSelection falla tres veces. La ampliación de la selección durante la exportación no pasa por teclado, sino por PTSL extend_selection_to_target_tracks(); la antigua función auxiliar Shift+Ö (_send_shift_oe) ya no está activa en la ruta de código actual.

# keys.py
def _send_key_to_app(pid, vk, cmd=False, shift=False):
    src = Quartz.CGEventSourceCreate(kCGEventSourceStateHIDSystemState)
    ev_dn = Quartz.CGEventCreateKeyboardEvent(src, vk, True)
    if cmd:
        Quartz.CGEventSetFlags(ev_dn, kCGEventFlagMaskCommand)
    Quartz.CGEventPostToPid(pid, ev_dn)
    Quartz.CGEventPostToPid(pid, Quartz.CGEventCreateKeyboardEvent(src, vk, False))

Constantes VK (códigos de hardware de macOS, independientes de la distribución): _VK_K=40 (Cmd+K = alternativa de Pre-Roll), _VK_F2=120 (renombrado en Interplay), _VK_ESC=53, _VK_RETURN=36. Los CGEvents se usan además en la automatización de UI de Interplay (F2/Tab/Cmd+A/C/V), ya que Interplay Access no tiene API de PTSL.

AppleScript / automatización de UI UI

AppleScript vía osascript (subprocess) se usa para acciones que no tienen API de PTSL:

  • Leer los títulos de ventana de PT (name of every window) — p. ej. para esperar al final del proceso de Consolidate
  • Lanzar el menú de exportación de PT (Clip → Export to MediaCentral). El Consolidate en sí se ejecuta vía PTSL consolidate_clip(), no por AppleScript.
  • Controlar Interplay Access (navegación con Tab, renombrado con F2, Cmd+C/V)
  • Leer el Static-Text de los cuadros de diálogo de PT (resultado de la exportación)
  • Activar la app (tell application "interplayAccess" to activate)

Todas las llamadas a AppleScript se encapsulan mediante _run_applescript(script, timeout=30), que arranca un proceso subprocess.run(). En caso de timeout, la función devuelve un código de retorno de error.

Permiso de accesibilidad: El acceso de AppleScript a System Events requiere que PunchBuddy (o Python) esté autorizado en Ajustes del sistema → Privacidad → Accesibilidad.

kqueue – monitorización de archivos macOS

kqueue es el mecanismo de notificación de eventos del kernel de macOS. PunchBuddy lo usa para vigilar el archivo de log de PT sin hacer polling:

# export.py – Simplificado
fd = os.open(log_path, os.O_RDONLY | os.O_NONBLOCK)
kq = select.kqueue()
ke = select.kevent(fd,
    filter=select.KQ_FILTER_VNODE,
    flags=select.KQ_EV_ADD | select.KQ_EV_ENABLE | select.KQ_EV_CLEAR,
    fflags=select.KQ_NOTE_WRITE | select.KQ_NOTE_EXTEND)

while time.time() < deadline:
    events = kq.control([ke], 4, min(remaining, 2.0))  # bloquea hasta un evento
    if events:
        # Leer bytes nuevos, comprobar el marcador
        ...

Ventajas frente al polling: reacción inmediata ante un evento de archivo (sin un intervalo fijo de 0.3s), sin carga de CPU entre escrituras. Alternativa a un polling de 0.3s si kqueue no está disponible por algún motivo.

API de control: Dispatcher + transportes API

En el núcleo hay una tabla de comandos command_dispatch(command, args) → (ok, msg) (_command_table()). UNA entrada por función (record_a/b, play, play_custom, goto_start, move_audio, import, todas las exportaciones, preset, vocaster_*, ping/status/list). Todos los transportes llaman exclusivamente a este dispatcher, de modo que HTTP y Socket comparten automáticamente el debounce y la serialización de la exportación.

Transporte 1 – HTTP: ThreadingMixIn + TCPServer embebido. Por cada petición de disparo, 200 OK inmediato; la función continúa en cola. Socket-Timeout (30 s) frente a clientes lentos. En un bind a 0.0.0.0 (LAN) se genera automáticamente un token (secrets.token_urlsafe(16)); las peticiones sin el token correcto reciben 401 (comparación segura frente a timing).

Transporte 2 – Unix-Domain-Socket: AF_UNIX/SOCK_STREAM en /tmp/punchbuddy.sock, permisos de archivo 0600. Sin pila IP → invisible para los filtros de red (Defender). Protocolo de líneas, bucle de accept con threads, cierre limpio vía atexit + manejadores de quit/señal + limpieza de socket obsoleto.

def command_dispatch(self, command, args=None):
    cmd = _COMMAND_ALIASES.get(command.lower(), command.lower())
    handler = self._command_table().get(cmd)
    if handler is None: return (False, f"unknown command: {command}")
    return handler(args)        # → (ok, msg)

Sistema de ajustes Config

Los ajustes se guardan como JSON en ~/.punchbuddy/settings.json. Al cargar se realiza un Deep-Merge recursivo con los valores predeterminados: las claves recién añadidas en el código se incorporan automáticamente y los valores de usuario existentes se conservan.

def _deep_merge(default, override):
    """Neue Keys aus default werden ergänzt; override gewinnt bei Konflikten."""
    if isinstance(default, dict) and isinstance(override, dict):
        out = dict(default)
        for k, v in override.items():
            out[k] = _deep_merge(default[k], v) if k in default else v
        return out
    return override

Las entradas de preset reciben además un merge de plantilla (_PRESET_TEMPLATE), para que las claves de preset recién añadidas aparezcan también en los presets antiguos ya guardados.
